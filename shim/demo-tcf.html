TCF → navigator.consent adapter

A proof-of-concept bridge between the IAB TCF v2.x CMP API and navigator.consent. One script, zero configuration.

What is this?

The TCF adapter is a single self-executing script (~3 KB minified) that transparently connects any IAB TCF v2.x compliant CMP to navigator.consent. It listens for TCF lifecycle events through the standard __tcfapi('addEventListener') API and registers purposes, vendors, and consent decisions with navigator.consent. No configuration required. No public API exposed. The CMP does not need to know the adapter exists.

How it works

  1. Polls for both __tcfapi and navigator.consent (100 ms interval, 10 s timeout). If either is missing, the adapter silently exits.
  2. Subscribes to TCF events via __tcfapi('addEventListener', 2, callback).
  3. On the first meaningful event (tcloaded, cmpuishown, or useractioncomplete):
    • Fetches the Global Vendor List via getVendorList (best-effort).
    • Calls registerInterface() with CMP identity from TCData.
    • Calls registerPurposes() with the 11 standard TCF purposes (consent and legitimate interest variants), plus special features.
    • Calls registerVendors() with vendor entries enriched from the GVL (names, privacy URLs, purpose mappings).
    • Calls updatePreferences() with the current consent state.
    • Calls requestConsent() to signal that a consent flow is active.
  4. On subsequent events (useractioncomplete): calls updatePreferences() with the updated consent state.

Timing

T0  CMP stub loads              window.__tcfapi = queuing function
T1  Adapter loads               polls for __tcfapi + navigator.consent
T2  Both found                  adapter calls addEventListener
T3  First event (tcloaded       adapter registers + syncs + requestConsent()
    or cmpuishown)
T4  useractioncomplete          adapter syncs updated consent

The adapter never blocks the CMP. All navigator.consent calls are asynchronous fire-and-forget.

Size and performance

The adapter is approximately 3 KB minified (under 1.5 KB gzipped). It makes zero network requests on its own and adds no runtime overhead beyond the initial registration. After the first TCF event is processed, the adapter is dormant, only waking when the CMP fires useractioncomplete.

Why this matters

      

        Privacy assistants (browser extensions that manage consent on behalf
        of the user) currently need CMP-specific integrations for every
        consent framework. With this adapter, any TCF-compliant CMP becomes
        automatically accessible through navigator.consent.
        Privacy assistants can read the purpose catalog, inspect vendor
        lists, and apply user preferences through a single, standard API.
      

      

        For CMPs, this requires zero integration effort. Drop the adapter
        script on the page (or let the browser/extension inject it), and the
        existing TCF implementation is bridged automatically.
