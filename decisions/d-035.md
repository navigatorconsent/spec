### D-035 registerInterface return surfaces presence and freshness

- **Question**: Where should the API surface the signals a CMP needs on first contact: whether an assistant is currently present, whether a previously stored catalog checksum exists, and the effective regulation context?
- **Why it matters**: The CMP's first call is `registerInterface()`. If presence and freshness are not on that return value, CMPs must either issue extra round-trips (one or more of `getRegulations()`, a new `ping()`, an enumeration call) or skip the optimization entirely. UserCentrics proposed a `ping()` method that subsumes regulations and adds an `agentStatus` enum. That literal shape would replace `getRegulations()` (settled by D-029), dropping the richer `source` and `browserDefault` fields in `RegulationInfo`.
- **Decision**: Augment the return shape of `registerInterface()`. Keep `getRegulations()` exactly as defined by D-029.
- **Decided behavior**:
  - `registerInterface()` returns:
    ```ts
    {
      registrationId: string;
      storedCatalogChecksum: string | null;
      assistantStatus: "absent" | "present";
      regulations: string[];
      jurisdiction: string | null;
    }
    ```
  - `storedCatalogChecksum` is the value previously written for this `cmp.id` and origin (see D-034), or `null` if none is stored. CMP compares it to the locally computed checksum and MAY skip `registerVendors()` and `registerPurposes()` on match.
  - `assistantStatus` reports whether at least one Privacy Assistant has called `registerAssistant()` for the current page context. Two values only: `"absent"` (no assistant currently registered) and `"present"` (one or more assistants registered). The browser MUST NOT disclose how many.
  - `regulations` and `jurisdiction` mirror the corresponding fields of `RegulationInfo` so CMPs do not need a follow-up `getRegulations()` call for the common case. The richer fields (`source`, `browserDefault`) are intentionally not duplicated. CMPs that need them call `getRegulations()`.
  - `getRegulations()` and `setRegulations()` (D-029) are unchanged. No `ping()` method is introduced.
- **Rationale**: gives CMPs a single round-trip discovery point at the natural moment they already make a call, avoids weakening D-029's `RegulationInfo` shape, and keeps the API surface small (no new top-level method).
- **Trade-offs accepted**:
  - The `regulations` and `jurisdiction` fields appear both on the registration return and on `RegulationInfo`. Documentation must clarify which to call when. Acceptable because the registration return is a one-time convenience for the common case, while `getRegulations()` covers the live, richer shape and is callable any time.
  - `assistantStatus` is binary by design. Use cases for "multiple" or "unknown" are deferred until concrete demand emerges.
  - Disclosing assistant presence to the page leaks the existence of an extension. The leak is intentional and bounded: only the binary signal is exposed, never identity, count, or capabilities, so the CMP can avoid pointless catalog payload work.
- **Follow-up changes**: RFC §6.4.1 (`registerInterface` signature and prose). Shim implementation, TypeScript types. Composes with D-034 for the checksum mechanism and with D-032 for the assistant-presence accounting.
