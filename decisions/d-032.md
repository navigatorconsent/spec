### D-032 Assistant self-registration replaces init()

- **Question**: Should Privacy Assistant Software register itself through a method symmetric to the CMP's `registerInterface()`, instead of the current lightweight `init()` signal?
- **Why it matters**: `init()` is a fire-and-forget readiness signal that returns no handle. Assistants have no stable identifier the browser can attach to provenance, audit records, or coordination logic. The asymmetry with `registerInterface()` (which mints a `registrationId`) makes assistant-side flows harder to reason about and limits future capability declarations.
- **Decision**: Replace `init()` with `registerAssistant()`. The browser mints an `assistantId` and returns it.
- **Decided behavior**:
  - `registerAssistant(metadata): Promise<{ assistantId: string }>` is extension-only.
  - `metadata` carries `vendor` (extension publisher, required), `version` (required), `displayName` (optional), and `frameworks` (optional `string[]` declaring supported compliance frameworks).
  - Browser mints `assistantId` and uses it as the stable handle for assistant-attributed provenance, event filtering, and future coordination methods.
  - Browser dispatches a `registerassistant` event (replacing the previous `init` event).
  - Browser appends a history entry of type `registerassistant` for each call.
  - Re-registration by the same extension MAY update metadata and SHOULD preserve the existing `assistantId` for continuity.
  - There is no inverse `unregisterAssistant()`. Browser detects extension teardown.
  - The previous `init()` method is removed entirely (no aliases, no legacy shim) since this draft is exploratory and carries no compatibility obligations.
- **Rationale**: gives the browser a stable identifier for the assistant, restores naming symmetry with `registerInterface()`, and creates a structured slot for capability declarations as the API evolves.
- **Trade-offs accepted**:
  - All existing call sites that referenced `init()` must update. Acceptable because the API is in draft and has no production deployments.
  - The handle returned by `registerAssistant()` is browser-minted, which means the assistant cannot pre-commit to a stable id across browser restarts unless the browser opts to persist mappings (left implementation-defined for v1).
- **Follow-up changes**: RFC §5.3 permission table, §6.3 `ConsentEventType` enum, §6.5 method list, §6.6.3 event ordering, Appendix A diagram. Shim implementation, TypeScript types, and `consent-record` / `event` JSON schemas.
