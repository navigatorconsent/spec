### D-033 Rename audit() to getHistory() and add listRegistrations()

- **Question**: How should the API surface the consent event history and current registration state, given that the existing `audit()` name causes confusion with regulatory or compliance audits?
- **Why it matters**: `audit()` returns a chronological event log (preference updates, withdrawals, hide/show, regulation changes, registration events). Readers conflate it with a privacy or compliance audit, which it is not. Separately, there is no method to enumerate currently-active registrations as a snapshot, so assistants must reconstruct state from the event log.
- **Decision**: Rename the event-log method to `getHistory()`. Rename the record type to `ConsentHistoryEntry`. Add a new `listRegistrations()` method that returns currently-active registrations as a snapshot.
- **Decided behavior**:
  - `getHistory(query?): Promise<ConsentHistoryEntry[]>` replaces `audit()`. Same query shape (`registrationId`, `from`, `to`, `limit`). Same extension-only permission boundary.
  - Internal record-type value `"audit"` (denoting the assistant queried the log) is renamed to `"query"` to avoid lexical recursion.
  - Internal record-type value `"init"` is renamed to `"registerassistant"` for consistency with D-032.
  - `listRegistrations(filter?): Promise<InterfaceRegistration[]>` returns the currently-active interface registrations visible in the calling context. Extension-only. Filter shape is `{ origin?, cmpId? }`. Where the structured `cmp` object replaces top-level `cmpId` (D-034), the filter accepts `cmp.id` instead.
  - Both methods MUST honor anti-spam controls and bounded response windows defined elsewhere in the RFC.
  - All RFC and shim references to "audit record" prose related to this method are reworded to "history entry". General prose about "audit trail" or "auditable" remains unchanged where it refers to the broader compliance concept.
- **Rationale**: removes the single biggest source of mental overlap with regulatory audits, while adding the missing snapshot capability so assistants do not need to scan the event log to enumerate active registrations.
- **Trade-offs accepted**:
  - Two surface changes in one decision record. Acceptable because both are renames or additions on the same conceptual area (visibility into stored state).
  - "History" is slightly heavier than "events", but reads more clearly as past-tense and avoids the DOM `Event` type ambiguity.
  - `listRegistrations()` returns the registration payloads as stored, not enriched with derived state; richer query methods are deferred.
- **Follow-up changes**: RFC §5.3 permission table, §6.3 `ConsentEventType` enum, §6.5 method list and prose, §10.2 anti-spam reference to bounded response windows. Shim implementation, TypeScript types, and `consent-record` / `event` JSON schemas.
