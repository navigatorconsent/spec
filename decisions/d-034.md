### D-034 Structured cmp metadata and opaque catalog checksum

- **Question**: How should `InterfaceRegistration` carry CMP identity and version metadata, and how should CMPs detect that a previously registered vendor and purpose catalog is still current?
- **Why it matters**: The current shape collapses CMP identity into a single required `versionIdentifier: string`. The name is ambiguous (SDK version, product version, customer config version) and pushes meaning into an opaque blob. Separately, CMPs re-register vendor and purpose catalogs on every page load with no mechanism to detect that the browser already holds an up-to-date copy. For large catalogs, this creates avoidable payload pressure.
- **Decision**: Replace `versionIdentifier` and the top-level `cmpId` with a structured `cmp` object. Add a separate optional opaque `catalogChecksum` to `InterfaceRegistration`. Browser stores the checksum opaquely and echoes it on registration so CMPs can decide whether to skip subsequent `registerVendors()` and `registerPurposes()` calls.
- **Decided behavior**:
  - `InterfaceRegistration.cmp` (object, required) carries:
    - `id` (optional, replaces the old top-level `cmpId`)
    - `version` (required, CMP product version)
    - `sdkVersion` (optional, integration SDK version)
    - `configVersion` (optional, customer configuration version, informational)
    - `language` (optional, BCP 47 default content language)
    - `displayName` (optional, human-readable label)
    - `frameworks` (optional `string[]`, declared compliance frameworks such as `tcf-v2.2`, `gpc`, `gpp`)
  - The previous required field `versionIdentifier` is removed. The previous top-level `cmpId` is removed (absorbed into `cmp.id`).
  - `InterfaceRegistration.catalogChecksum` (optional `string`) is an opaque CMP-computed digest covering both vendors and purposes together. The browser MUST store the value verbatim alongside the registration and MUST NOT interpret, hash, or validate it. Hashing strategy is entirely the CMP's choice.
  - Storage scope for `catalogChecksum` is per origin per `cmp.id`. Where `cmp.id` is absent, scope is per origin per `vendor`.
  - Browser invalidates stored checksums on user clear-state and on RFC schema-version bumps. CMPs SHOULD treat any other invalidation as best-effort.
  - The previously-stored value is surfaced to the CMP through the `registerInterface()` return shape (see D-035) so the CMP can compare and skip catalog re-registration when matching.
  - Per-catalog checksums (separate values for vendors and purposes) are out of scope for v1.
- **Rationale**: structured fields make the metadata self-documenting and useful for assistant telemetry and dev-tools display. Keeping freshness as a separate opaque field decouples "who is registering" from "is the catalog cached", lets the CMP pick any hashing strategy, and matches the UserCentrics proposal in shape while preserving the richer identity object.
- **Trade-offs accepted**:
  - Two related fields (`cmp.configVersion` and `catalogChecksum`) cover overlapping concerns. Documentation must be explicit that `configVersion` is informational and that `catalogChecksum` is the freshness signal.
  - Splitting `cmp` from the top level adds nesting. Acceptable for the clarity gain.
  - One combined checksum cannot signal "vendors changed but purposes did not". Per-catalog granularity is deferred to a later decision if needed.
- **Follow-up changes**: RFC §6.3 (`InterfaceRegistration`), §6.4.1 (`registerInterface` prose), §6.4.2 and §6.4.3 (note skip path), §10.6 (re-registration prose currently keyed on `cmpId` becomes `cmp.id`). Shim implementation, TypeScript types, `cmp.schema.json`. Composes with D-035 for the surfaced freshness value.
